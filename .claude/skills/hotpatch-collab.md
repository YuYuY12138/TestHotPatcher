# UE HotPatch 项目协作规范

本 skill 定义了在该项目中与用户高效协作的工作方式。每次收到需求时必须遵守以下规则。

## 1. 需求对齐：先问后做

收到需求后，如果存在以下情况，**必须先提问确认，不能直接动手**：
- 目标不明确（不知道改哪里、改成什么样）
- 存在多种实现方案需要取舍
- 改动可能影响现有功能或工具链
- 涉及策划/美术使用的工具，需要考虑使用者体验

提问格式要简洁，列出具体的不确定点，等用户确认后再执行。

## 2. 改动说明：每次修改必须说清楚

每次修改代码或文件后，必须按以下格式说明：

- **改了哪里**：文件名 + 行/函数
- **改了什么**：简述变更内容
- **为什么这么改**：技术原因或设计理由
- **影响范围**：是否影响其他模块、平台、工具流程

这样用户可以有效 Review，不会出现"改了什么不知道"的情况。

## 3. 代码注释规范

所有写出的 C++ 代码必须包含注释：
- **类级别**：职责、生命周期、使用方式、重要约束（特别是"禁止"类）
- **函数级别**：参数含义、返回值、执行步骤、调用时机
- **关键逻辑行内**：解释"为什么"，不是重复"做了什么"
- 注释语言：中文为主，英文 bat/sh 脚本用英文（避免编码问题）

## 4. 宏观影响评估：子问题不能割裂看

解决局部问题时，必须考虑对整体的影响：
- 修改一个配置字段，评估对 Release 和 Patch 两个流程的影响
- 修改 MountPoint 路径，同时检查 Release/Patch 两个 JSON 是否需要同步
- 改 Android 打包配置，考虑是否影响 Windows 打包
- 修改 HotPatcher 设置，考虑基础包和补丁包是否需要重新生成

## 5. 工具视角：站在使用者角度设计

本项目的热更工具（push_patch.bat、HotPatcher JSON 配置等）的直接用户是**策划**，不是程序员。设计和评估时需要同时考虑两个视角：

**程序员视角**：技术是否正确、是否有潜在问题

**策划/工具使用者视角**：
- 步骤是否简单，能否一键完成
- 出错时提示是否清晰，知道下一步怎么做
- 是否容易误操作（如版本号写错、路径配错）
- 能否尽量自动化，减少手动填写

每次提出工具方案时，主动评估：这个流程对策划来说是否足够简单？有没有容易出错的环节？能否进一步自动化？

## 6. 当前项目背景（快速上下文）

- 引擎：UE 5.7 + UnLua + HotPatcher
- 目标平台：Android
- 热更内容：UE 资产（uasset/umap）+ Lua 脚本（Script/）
- 热更流程：HotPatcher 出 patch → push_patch.bat 推送 → 游戏启动 → LaunchMap → 点按钮 → MountPak → 切 TestMap
- Lua 搜索路径：`Script/?.lua`（相对于 ProjectDir）
- patch 存放路径：`Saved/Paks/`（手机上引擎自动扫描）
- pending 路径：`Saved/HotUpdate/Pending/`（push_patch.bat 推送到这里）
- bIoStore：False（必须关闭，否则传统 pak 热更不生效）
- Bootstrap 关卡（LaunchMap）严禁绑定 UnLua，否则 Lua VM 提前初始化热更失效
