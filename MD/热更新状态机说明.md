## 热更新状态机说明

### 状态定义与流转

热更新系统采用状态机驱动，UHotUpdateSubsystem 在整个更新过程中维护当前状态，每个状态对应一个明确的职责阶段。以下按实际执行顺序逐一说明。

### None

系统初始状态。GameInstance 创建后、UHotUpdateSubsystem 尚未开始工作时处于此状态。此时不做任何操作，等待 Bootstrap 关卡 BeginPlay 后触发启动。

### Initializing

Subsystem 开始初始化。这个阶段负责创建和检查本地工作目录（Saved/HotUpdate/Pending、Saved/Paks 等），读取本地 version.json 获取当前版本号，以及准备网络模块。如果本地目录不存在则创建，如果 version.json 不存在则视为首次运行，以 APK 编译时的版本号作为当前版本。初始化完成后自动进入下一状态。

### RecoveringPendingPatch

检查上次是否有未完成的下载任务。扫描 Pending 目录，如果发现有残留的 .download 临时文件和对应的任务状态记录，说明上次下载被中断（用户杀进程、手机重启等）。此阶段只做本地状态恢复，不发起网络请求。恢复的信息包括上次下载的目标版本号、已下载的文件大小、文件路径等，供后续与服务端清单比对时决定是续传还是重新下载。如果没有残留任务则直接跳过。

### RequestingManifest

向 Java 服务端发起版本检查请求。通过 HTTP GET 请求版本清单 API，携带本地的主版本号、资源版本号、热更版本号和平台信息。等待服务端返回最新版本信息、补丁下载地址、文件大小和 MD5 等。如果网络请求失败（超时、断网等），可以按策略重试若干次。如果重试仍然失败，根据产品策略决定是跳过更新用旧版本进入游戏，还是提示用户检查网络。

### ComparingVersion

拿到服务端清单后，与本地版本逐段比较。这个阶段的输出是一个明确的更新决策：无需更新、需要热更补丁、需要重装基础包、或需要应用商店更新。如果上一步恢复了未完成任务，这里还要判断未完成任务对应的版本是否仍然有效——如果服务端已经发布了更新的版本，则丢弃旧的未完成任务。比较完成后，根据结果决定下一步：无更新则直接跳到 Finalizing，需要重装则提示用户跳转下载页，需要热更则进入等待确认。

### WaitingForUserConfirm

非强制更新时，在更新 UI 上展示更新信息（版本号、更新内容说明、补丁大小），等待用户点击确认。用户可以选择"立即更新"进入下载流程，或选择"稍后"跳过更新进入旧版本游戏。如果服务端标记了 forceUpdate，则界面上不展示"稍后"按钮，用户必须更新才能继续。用户确认后进入下一状态。

### PreparingDownload

下载前的准备工作。检查设备剩余存储空间是否足够（建议要求补丁大小的 1.5 倍以上的可用空间），如果不足则提示用户清理。检查当前网络类型，如果从 WiFi 切换到了移动数据，可以弹窗让用户确认是否继续。如果上一步恢复了未完成任务且版本仍有效，这里准备断点续传的参数（已下载大小、Range 起始位置）。准备完毕后进入下载。

### Downloading

调用 DTKit 的 UDownloadProxy 执行实际的 HTTP 文件下载。这是整个流程中用户等待时间最长的阶段，更新 UI 上展示下载进度条、已下载大小、下载速度。DTKit 负责 HEAD 请求获取文件大小、Range 分片下载、边下载边计算 MD5。下载过程中文件以 .download 后缀存储在 Pending 目录中，防止半截文件被误识别为有效补丁。下载过程中可以被用户暂停（进入 Paused 状态）或被网络中断暂停。下载完成后自动进入校验阶段。

### Paused

下载暂停状态。可以由用户主动暂停触发，也可以由网络断开、切到后台等事件触发。暂停时 DTKit 停止当前 HTTP 请求，但保留已下载的数据和 MD5 计算的中间状态（进程内）。用户点击继续或网络恢复后，回到 Downloading 状态，DTKit 从断点继续下载。如果暂停期间进程被杀死，下次启动时由 RecoveringPendingPatch 阶段恢复。

### Verifying

下载完成后对文件进行校验。首先检查文件大小是否与服务端清单一致，然后比对 MD5。如果 DTKit 在下载过程中已经计算了完整文件的 MD5，直接取用；如果是断点续传且跨越了进程重启，则需要重新从文件头流式读取整个文件计算 MD5。校验失败时删除已下载文件，回到 Downloading 状态重新下载，最多重试 3 次。3 次仍失败则进入 Failed 状态。校验通过后进入安装阶段。

### Installing

将校验通过的补丁文件从 Pending 目录移动到正式目录。如果采用全量补丁策略，在移入新补丁之前不要删除旧补丁——先完成移动，确保新文件落盘成功后，再清理旧补丁文件。文件移动操作本身可能因磁盘错误失败，失败时保持旧版本可用，进入 Failed 状态。移动成功后进入挂载阶段。

### Mounting

运行时挂载补丁 Pak。调用 UFlibPakHelper::MountPak 将补丁以高于基础包的优先级挂载到引擎的文件系统中。挂载后还需要处理 Shader Library（调用 LoadShaderbytecode 加载补丁中的着色器编译产物）和 Asset Registry（调用 LoadAssetRegistry 合并增量资源注册表，使引擎能发现补丁中新增的资源）。如果补丁不包含 Shader 或新增资源类型，对应的处理可以跳过。挂载失败时不提交版本，进入 Failed 状态，旧版本仍然可用。挂载成功后进入最终确认。

### Finalizing

更新的最终提交阶段。将新版本号写入 version.json，清理 Pending 目录中的临时文件，删除旧补丁 Pak（全量补丁策略下），更新启动保护标记。这是整个更新事务的提交点——只有走到这一步，本次更新才算正式生效。Finalizing 完成后进入 Completed。

如果整个流程中没有更新（ComparingVersion 阶段判定无更新），也会跳到 Finalizing，此时只是标记"检查完毕"，不做任何文件操作，然后进入 Completed。

### Completed

热更新流程正常结束。此时通知 Bootstrap 关卡：热更完成，可以初始化业务 Lua、加载正式登录关卡、销毁更新 UI。Subsystem 本身保持常驻（跟随 GameInstance 生命周期），记录当前版本和挂载信息，供后续可能的后台预下载或运行时查询使用。

### Failed

更新过程中出现不可自动恢复的错误。可能的触发场景包括：下载重试 3 次仍失败、文件系统写入错误、Pak 挂载失败、Shader Library 加载失败等。进入此状态时，旧版本必须保持可用。更新 UI 上展示错误信息和重试按钮。用户点击重试后，根据失败发生的阶段回到对应状态重新执行（例如下载失败回到 Downloading，挂载失败回到 Mounting）。如果产品策略允许，也可以提供"跳过更新"选项让用户用旧版本进入游戏。

### Canceled

用户主动取消更新。在 WaitingForUserConfirm 阶段选择"稍后"，或在下载过程中选择"取消"时进入此状态。取消后清理 Pending 目录中的临时文件（或保留供下次启动续传，视产品策略而定），然后用当前本地版本正常进入游戏。

### 状态流转总结

正常完整流程：None → Initializing → RecoveringPendingPatch → RequestingManifest → ComparingVersion → WaitingForUserConfirm → PreparingDownload → Downloading → Verifying → Installing → Mounting → Finalizing → Completed

无更新的快速路径：None → Initializing → RecoveringPendingPatch → RequestingManifest → ComparingVersion → Finalizing → Completed

用户跳过更新：ComparingVersion → WaitingForUserConfirm → Canceled

下载中暂停恢复：Downloading → Paused → Downloading

校验失败重试：Verifying → Downloading → Verifying

不可恢复错误：任意阶段 → Failed
