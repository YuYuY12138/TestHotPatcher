# G01 Android 热更新完整流程设计

---

## 一、版本号体系

```
主版本号 . 资源版本号 . 热更版本号
   │           │           │
   │           │           └── 热更迭代（每次出补丁 +1，不需要重装）
   │           └────────────── 资源大版本（需要新基础包时 +1，需要重装）
   └────────────────────────── 应用大版本（重大更新，需要重新上架应用商店）

```

### 版本号示例

```
1.0.0    首包（基础包），上架应用商店
1.0.1    第 1 次热更（补丁 Pak，不重装）
1.0.2    第 2 次热更（补丁 Pak，不重装）
1.0.5    第 5 次热更（补丁 Pak，不重装）
1.1.0    资源大版本更新（新基础包，需要重装 APK）
1.1.1    基于 1.1.0 的第 1 次热更
2.0.0    应用大版本，重新上架
```

### 三种更新场景

| 场景 | 版本变化 | 更新方式 | 用户体验 |
|------|---------|---------|---------|
| **热更新** | 1.0.1 → 1.0.2 | 下载补丁 Pak → 挂载 → 直接进入游戏 | 等待几秒~几分钟，无需重启 |
| **资源大版本** | 1.0.x → 1.1.0 | 需要重新下载安装 APK + OBB | 跳转应用商店或下载页 |
| **应用大版本** | 1.x.x → 2.0.0 | 需要重新下载安装 APK | 跳转应用商店 |

## 三、关键设计决策：为什么不需要重启

```
传统方案（需要重启）：
  引擎启动 → Lua 初始化 → 进入游戏 → 检查更新 → 下载 → 安装
  → 此时 Lua 已经加载了旧版本 → 必须重启才能加载新版本

G01 方案（不需要重启）：
  引擎启动 → ★热更检查（C++ 驱动）→ 下载 → MountPak
  → Lua 初始化（此时 Pak 已挂载，首次 require 拿到的就是新版本）
  → 进入游戏

核心原则：热更新检查必须发生在 UnLua 初始化之前。
         只要 Lua VM 还没 require 任何脚本，
         挂载补丁 Pak 后 Lua 自然读取到最新版本。
```

**更新界面为什么用 C++ UI 而不是 Lua UI**：

```
热更检查发生在 UnLua 初始化之前 → Lua 还不可用
→ 更新提示、进度条等 UI 必须用 C++ / 原生 UMG 实现
→ 不依赖 Lua 框架（UIManager 此时尚未初始化）

这个 C++ UI 只需要很简单的功能：
· 一个弹窗（版本信息 + 确认按钮）
· 一个进度条
· 几行文字
不需要复杂的 UI 框架
```

### 4.2 下载失败处理

```
下载过程中可能遇到的问题：

网络断开
  → 自动暂停 → 监听网络恢复 → 自动继续
  → 文件已下载部分保留，支持断点续传

下载超时
  → 自动重试（最多 3 次）
  → 每次重试从断点继续

校验失败
  → 删除已下载文件
  → 重新下载（最多 3 次）
  → 3 次仍失败 → 提示用户"网络异常，请检查网络后重试"

存储空间不足
  → 下载前预检查（需要 补丁大小 × 1.5 的剩余空间）
  → 不足 → 提示用户清理空间

WiFi → 移动数据切换
  → 暂停下载 → 弹窗"当前使用移动数据，是否继续？"
```

### 4.3 多个补丁的下载策略

```
场景：用户从 1.0.0 直接更新到 1.0.3

方案 A：全量补丁（推荐）
  服务端直接下发一个基于 1.0.0 生成的全量补丁
  包含 1.0.1 + 1.0.2 + 1.0.3 的所有变更
  用户只需下载一个 Pak
  
  优点：下载次数少，逻辑简单
  缺点：补丁体积较大

方案 B：增量补丁
  按顺序下载 1.0.1.pak → 1.0.2.pak → 1.0.3.pak
  串行下载，每个下载完校验后再下一个
  
  优点：每个补丁体积小
  缺点：版本链管理复杂，中间版本不能跳过

建议：初期用方案 A（全量），简单可靠。
     等补丁体积成为瓶颈后再改方案 B。
```

## 五、版本检查 API 设计

### 请求

```
GET /api/v1/check-update
  ?appVersion=1.0.0          ← APK 版本（编译时写死）
  &patchVersion=1.0.2        ← 当前补丁版本（本地 version.json）
  &platform=android
```

### 响应

```json
{
  "code": 0,
  "data": {
    "updateType": "patch",
    "latestPatchVersion": "1.0.3",
    "minAppVersion": "1.0.0",
    "forceUpdate": false,
    "releaseNote": "1. 修复战斗结算异常\n2. 新增春节活动",
    "patches": [
      {
        "version": "1.0.3",
        "pakUrl": "https://cdn.xxx.com/patches/1.0.3_Android_ASTC_001_P.pak",
        "pakSize": 15728640,
        "md5": "a1b2c3d4e5f6..."
      }
    ]
  }
}
```

### 更新类型判定逻辑（服务端）

```
if (客户端 appVersion < minAppVersion)
    → updateType = "forceInstall"    // 必须重装 APK
    
else if (客户端 patchVersion < latestPatchVersion)
    → updateType = "patch"           // 热更新
    → forceUpdate = true/false       // 是否强制
    
else
    → updateType = "none"            // 无更新
```

## 六、本地版本文件（version.json）

```json
{
  "appVersion": "1.0.0",
  "patchVersion": "1.0.2",
  "installedPatches": [
    {
      "version": "1.0.1",
      "pakFile": "1.0.1_Android_ASTC_001_P.pak",
      "md5": "a1b2c3...",
      "installTime": "2026-06-15T10:30:00Z"
    },
    {
      "version": "1.0.2",
      "pakFile": "1.0.2_Android_ASTC_001_P.pak",
      "md5": "d4e5f6...",
      "installTime": "2026-06-16T08:00:00Z"
    }
  ]
}
```

存储位置：`Saved/HotUpdate/version.json`

- `appVersion`：APK 编译时写入，不随热更变化
- `patchVersion`：每次安装补丁后更新
- `installedPatches`：已安装补丁记录，用于启动时校验和回滚

## 七、启动崩溃保护

```
正常启动流程：
  write(launchFlag = "starting")
  → 挂载补丁 Pak
  → 初始化成功
  → write(launchFlag = "running")

崩溃检测：
  启动时读取 launchFlag
  
  if (launchFlag == "starting" 连续 2 次)
      → 说明上次启动崩溃了
      → 卸载最新补丁（从 Saved/Paks/ 删除）
      → 回退 version.json
      → 上报崩溃信息到服务器
      → 用旧版本继续启动

  这样即使某个补丁有 bug 导致崩溃，
  用户最多经历 2 次闪退后自动恢复。
```

## 八、总结：三种场景对比

| | 热更新 | 资源大版本 | 应用大版本 |
|---|---|---|---|
| 版本号变化 | 1.0.x → 1.0.y | 1.0.x → 1.1.0 | 1.x.x → 2.0.0 |
| 更新内容 | Lua 脚本 + UE 资源 | C++ 代码变更 / 大量资源重构 | 引擎升级 / 重大架构变更 |
| 更新方式 | 下载补丁 Pak | 重新下载安装 APK+OBB | 应用商店更新 |
| 需要重启？ | **不需要** | 需要重装 | 需要重装 |
| 用户感知 | 启动时等几秒~几分钟 | 跳转下载页 | 跳转应用商店 |
| 服务端配合 | 版本 API + CDN 托管 Pak | 版本 API + APK 下载链接 | 应用商店审核 |
